# 共同編集の運用方針（提案）

> このLP（`food_photo`）を **2人**で安全に共同編集するための運用ルールの提案です。
> お互いAIコーディングツール（**Codex / Claude Code** など）を使って編集する前提で書いています。
> 内容に合意いただけたら、この提案をマージして運用開始としましょう。気になる点はPRのコメントでどうぞ。

---

## 0. 前提

- **正本（唯一の最新）= GitHubリポジトリ `electricdreams1987/food_photo` の `main` ブランチ**
- 手元のコピーやチャットに貼ったコードは「作業用」。**正本はいつもリモートのmain**、という認識を揃えます。
- メンバー：2名（リモートで連絡を取りながら作業）

## 1. 基本フロー：`main`に直接pushしない。ブランチ＋PRで進める

```
① 最新のmainを取り込む      (pull)
② 作業用ブランチを切る       (例: fix-price)
③ ブランチ上で編集・コミット
④ ブランチをpushしてPRを作る
⑤ 相手が確認 → mainにマージ
```

**なぜ直接pushにしないか**
- 同じ行を触ったときに、片方の変更が消える／pushが弾かれる事故を防ぐため
- PRにするとGitHubが**コンフリクト（衝突）を自動検出**してくれる
- マージ前に相手が中身を確認できる（＝品質チェック）
- `git push -f`（force push）の出番がなくなる ← 一番怖い事故の元を断つ

## 2. AIツールでのやり方（コマンドは覚えなくてOK）

Codex でも Claude Code でも、日本語（自然文）でこう頼めば裏でgitを実行してくれます。

| 頼むこと | 実行される内容 |
|---|---|
| 「最新のmainを取り込んで、`fix-price` ブランチを作って」 | `git switch main && git pull` → `git switch -c fix-price` |
| （編集を依頼） | ファイル編集 |
| 「コミットして、このブランチをpushしてPRを作って」 | `git commit` → `git push -u origin fix-price` → `gh pr create` |
| 「PRをマージして」 | `gh pr merge --squash`（マージ後ブランチ削除） |

> ブランチ名は内容が分かる短い英語で：`fix-price` / `hero-photo` / `update-copy` / `fix-typo` など。

## 3. ルール（2人運用の要点）

1. **作業前に必ず最新から**：ブランチは必ず最新の `main` から切る（古い状態から切ると衝突の元）。
2. **1タスク＝1ブランチ＝1PR、小さく**：「料金だけ」「ヒーロー写真だけ」と粒度を小さく。大改修を1本にまとめない。
3. **着手を宣言する**：「今から◯◯セクション触ります」と一声。**同じ箇所の同時編集を避ける**のが最大の衝突予防。
4. **PRはこまめにマージ**：枝を長く放置すると本体と乖離して衝突が増える。**当日〜翌日中**のマージが理想。
5. **force push 禁止**：共有ブランチ・mainに対して `git push -f` は絶対にしない。
6. **マージ前にプレビュー確認**：ブラウザで表示＋スマホ幅（幅375px）を確認してからマージ。
7. **コミットメッセージは1行で「何を変えたか」**：例 `料金表を3プランに変更` / `ヒーローをsteak-glamに差し替え`。

## 4. コンフリクト（衝突）が起きたら

PR画面に「This branch has conflicts」と出たら、慌てず、使っているAIツールに：

> **「mainの最新を取り込んで、コンフリクトを解消して」**

と頼めば、`git pull origin main` してマージ処理をしてくれます。
「どちらの変更を残すか」の判断が要る箇所だけ、人間が決めればOKです。

## 5. デザイン方針・差し替え箇所・プロンプト集

編集時のデザイン方針（配色・フォント）、差し替え箇所リスト、そのまま使えるプロンプト集は
`README.md` にまとめてあります。編集前に一読を。方針をブラさないため、AIへの依頼には
**「デザイン方針は維持して」**を一言添えるのがおすすめです。

## 6.（任意）mainのブランチ保護

「うっかりmainに直push」を仕組みで防ぎたい場合、GitHubの設定で **main のブランチ保護**
（直接pushを禁止・PR経由のみ許可）を有効にできます。2人なら必須ではありませんが、
事故を完全に防ぎたいなら有効です。導入するかは相談で決めましょう。

---

### まとめ（この1枚でやること）
**最新main → ブランチ → 編集 → push → PR → 相手が確認 → マージ。** これだけ。
迷ったら「直pushせずブランチ」で固定 = 判断コストが減り、事故が消えます。

この方針でよければ、このPRをマージして運用開始としましょう 🙌
